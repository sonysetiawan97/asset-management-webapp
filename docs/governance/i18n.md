# Internationalization (i18n)

## Setup

The app uses **i18next** with two plugins:

| Plugin | Purpose |
|---|---|
| `i18next-browser-languagedetector` | Auto-detects browser language on first load |
| `i18next-http-backend` | Lazy-loads locale files from `/locales/{{lng}}/{{ns}}.json` |

**Configuration file**: `src/utils/i18n.ts`

```ts
i18n
  .use(HttpBackend)
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: "en",
    supportedLngs: ["en", "id"],
    nonExplicitSupportedLngs: true,
    debug: VITE_APP_ENV === "development",
    interpolation: { escapeValue: false },
    backend: { loadPath: "/locales/{{lng}}/{{ns}}.json" },
    ns: ["common", "i18n"],
    defaultNS: "common",
  });
```

## Supported Languages

| Code | Language | Locale File |
|---|---|---|
| `en` | English | `public/locales/en-US/common.json` |
| `id` | Indonesian | `public/locales/id/common.json` |

**Note**: The locale files are in `public/locales/`, NOT `src/locales/`. They are served as static files and loaded via HTTP.

## Namespaces

Two namespaces are configured:

| Namespace | File | Purpose |
|---|---|---|
| `common` | `common.json` | Main translation file (all UI strings) |
| `i18n` | `i18n.json` | Reserved for future use |

The default namespace is `common`, so you don't need to specify it in most cases.

## Translation Key Patterns

Follow these conventions for organizing translation keys:

### Module Pages

```json
{
  "modules": {
    "<module_name>": {
      "list": { "title": "Page Title" },
      "create": { "title": "Create Title", "notification": { "success": "Created!" } },
      "read": { "title": "Detail Title" },
      "update": { "title": "Edit Title", "notification": { "success": "Updated!" } },
      "form": { "field_name": "Field Label", "another_field": "Another Label" }
    }
  }
}
```

### Sidebar Menu

```json
{
  "sidebar": {
    "<section>": {
      "menu": {
        "<item_name>": "Display Name"
      }
    }
  }
}
```

### Common UI Elements

```json
{
  "common": {
    "actions": { "create": "Create", "edit": "Edit", "delete": "Delete", "back": "Back" },
    "status": { "active": "Active", "inactive": "Inactive" }
  }
}
```

## Usage in Components

```tsx
import { useTranslation } from "react-i18next";

const MyComponent = () => {
  const { t } = useTranslation();

  return (
    <div>
      <h1>{t("modules.users.list.title")}</h1>
      <p>{t("modules.users.form.username")}</p>
    </div>
  );
};
```

## Rules

1. **All user-facing strings must use `t()`** — no hardcoded text in JSX
2. **Use the full key path** — e.g., `t("modules.users.list.title")` not `t("title")`
3. **Keep keys flat** — use dot notation, not nested objects in keys
4. **Add keys to both `en-US/common.json` and `id/common.json`** — maintain parity
5. **Use interpolation for dynamic values**:
   ```tsx
   t("modules.users.form.welcome", { name: user.name })
   // In JSON: "welcome": "Welcome, {{name}}!"
   ```

## Changing Language

The language is auto-detected from the browser on first load. Users can change language via a language switcher (if implemented). The selected language is stored in `localStorage` under `i18nextLng`.

## Sign-in Error Fallback

The i18n key `signin.notification.failed` (`"Invalid username or password"`) is now used only as a **fallback** when the backend does not return an error message. The primary error message comes from the backend response body (`response.data.message`):

```tsx
// Signin.tsx
const msg = (err as Error).message || t("signin.notification.failed");
```

This means the displayed message dynamically reflects the actual auth failure reason (wrong password, inactive user, or deleted account).
